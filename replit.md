# HakaseAI — BioDigital Twin Platform

## Overview
HakaseAI is a BioDigital Twin Platform designed to simulate **pre-clinical** drug discovery and development across four layers (May 2026 restructure): **L1 In Vitro → L2 Confirmation (in-silico ↔ ex-vivo sub-tracks) → L3 Animal Cohort → L4 First In Human**. The previous April 2026 ordering (`In Vitro / In Silico / Ex Vivo / Animal Cohort`) has been collapsed: the existing `Layer2InSilicoScaffold` and `Layer3ExVivoScaffold` now live as switchable sub-tracks inside the new L2 Confirmation wrapper (gated by `confirmationSubTrack` state in `HakaseAI.tsx`); the Animal Cohort body and runner have been renumbered from slot 4 to slot 3 (all `simStatus` / `mergeLayerFlags` / `mergeLayerProvenance` / `getFlagsForLayer` / `safetyOverrides.toLayer` / `ProvenanceFooter layer={…}` literals remapped 4 → 3, and `provenanceL4` reads in the active path swapped to the existing `provenanceL3` accessor); slot 4 now hosts `FirstInHumanPanel` (defined at end of `HakaseAI.tsx`) which derives MABEL / starting-dose / escalation seeds / IND-readiness purely from `animalResults` (no fabricated data) and gracefully empty-states when `simStatus[3] !== "complete"`. **HAIOps.tsx and safetyFlags.ts were not modified** per long-standing project policy. Patient Cohort and Trial Digital Twin have been moved out of pre-clinical scope and are now handed to Hakase Clinical via API per Section 8 of the architecture doc. Their source code is preserved but rendered unreachable via the `HIDE_PATIENT_TRIAL_LAYERS = true` flag in `artifacts/hakase-ai/src/lib/layerArchitecture.ts` ("hide, don't delete"). The platform aims to accelerate drug development by providing comprehensive in-silico predictions and analyses, integrating computational chemistry, PK/PD modeling, and AI-driven insights to guide decision-making from compound identification to FIH dose projection.

### Zero-Lab Pivot (May 13, 2026 — supersedes prior IVIVE chip-vendor strategy)
HakaseAI is marketed as a **100% Zero-Lab In-Silico Cascade**. The customer-facing pipeline is software-only end-to-end (SMILES → L4 FIH readiness) with no physical assays, no tissue chips, no organoids in the run path. Implications:
- **Ex-Vivo (L2 sub-track) is computational tissue simulation** — virtual multi-cellular tissue environment derived from GTEx tissue-expression baselines + Tabula Sapiens v2 cell-type atlas + DepMap PRISM, NOT physical organoids/PCLS/organ-on-chip data ingestion.
- **IVIVE recalibration substrate is literature + public PK datasets + virtual-tissue outputs ONLY**. The May 6 plan to use Emulate / CN Bio / Mimetas chip-vendor data as recalibration substrate is **deprecated**. Do not reintroduce wet-lab assay terminology ("organoid panels", "primary hepatocyte CLint", "hERG patch-clamp", "microsomal stability", "perfusion", "assay telemetry", "CRO organoid panels") into customer-facing marketing or scaffold UI labels.

### Tier ladder canonical names (May 13, 2026)
The collapsed `MEASURED` tier has been split into three regulatorily-distinct tiers. Use these names verbatim across marketing, user guide, and (after the schema rewrite lands) in-app badges. They are the standard biotech / DMPK terminology — do not invent SaaS-flavoured synonyms like "CUSTOMER" or engineery names like "AUTO".

**Computed-side** (platform-generated): `HEURISTIC` / `STRUCTURAL` / `ML`.
**Substrate-side** (external evidence): `LITERATURE` / `IN-HOUSE` / `CLASS-MEDIAN`.
- `LITERATURE` — published value with DOI / PMID for this compound or a Tanimoto ≥ 0.4 congener.
- `IN-HOUSE` — bench / CRO measurement supplied by the operating team with assay method + lab + date.
- `CLASS-MEDIAN` — Lombardo 2018 / Smith 2010 drug-class anchor. Runnable fallback only.

**Cross-species defensibility rule:** for fu_p AND B/P, human + ≥1 preclinical species each must be `LITERATURE` or `IN-HOUSE`. `CLASS-MEDIAN` does NOT satisfy.

**Substrate mode** (sealed in IVIVE sign-off + propagated to PreClinicalPackage): `literature` / `in-house` / `class-median` / `mixed`.

**In-app badge transition:** the dashboard IVIVE panel still shows the legacy `MEASURED` badge for both LITERATURE and IN-HOUSE inputs; the per-cell schema split is a planned follow-up (consultant proposal #2 / #6). Marketing surfaces and the user guide already use the new ladder.

## System Architecture

### UI/UX Decisions
The platform features a dashboard (`HakaseAI.tsx`) with a dark color scheme (`bg-[#07080c]`) and uses the Inter font. Each simulation layer is color-coded under the new ordering: **Violet for L1 (In Vitro), Cyan for L2 (In Silico), Amber for L3 (Ex Vivo), Emerald for L4 (Animal Cohort)**. ### Technical Implementations
The project is a pnpm monorepo using TypeScript (v5.9) and Node.js (v24). The backend API server uses Express 5, PostgreSQL with Drizzle ORM, and Zod for validation. API codegen is handled by Orval from an OpenAPI specification. Frontend is a React + Vite SPA.

#### HAIOps Framework
The HAIOps framework (`Healthcare AI Operations`) integrates safety and governance. It includes a Safety Flag Cascade engine (cardiac, hepatotoxicity, DDI, narrow-margin, high-grade-ae, subgroup-disparity) that propagates across all simulation layers. Overrides are logged, and provenance metadata (engine version, timestamp) is displayed.

#### Source-Quality Scoring & Cascade Gate (context-independent foundation)
Every L1 output is tagged with a `Quality` record (`artifacts/hakase-ai/src/lib/sourceQuality.ts`) carrying a tier — `experimental` (1.0) / `enriched` (0.7) / `heuristic` (0.3) / `inferred` (0.1) — plus optional ChEMBL measurement payload (n, distinctCompounds, range_nM, geometricMean_nM, species, assayTypes, assayFormats, derived confidence). The API route `GET /api/chembl/similar-targets/v2` (`artifacts/api-server/src/routes/chembl.ts`) aggregates the full payload per target; v1 is preserved for back-compat. The frontend client `chemblTargetLookup.ts` calls v2 first and falls back to v1. Each `TargetSuggestion` (`targetSuggestion.ts`) carries a `quality` field, and the L1 target-suggestion cards render n / range / dominant assay type / species and gray-out the card with "Insufficient data — exploratory" when `shouldSuppressEstimate` returns true. The cascade gate (`artifacts/hakase-ai/src/lib/cascadeGate.ts`) enforces three rules before L1→L2 auto-cascade is permitted: (a) compound is not NO-GO, (b) every safety-critical output (hERG, hepatotoxicity, bioactivation, CYP panel) meets minimum quality ≥ 0.6, and (c) average non-safety quality ≥ 0.6. The `evaluateCascadeGate` memo in `HakaseAI.tsx` drives a Stage-Gate banner inside the L1→L2 Cascade Preview section that either confirms auto-cascade, or shows blocked outputs and exposes an "Override & advance to L2" button which logs to the HAIOps provenance trail via `recordCascadeOverride` + `addOverride`. **Constraint**: `safetyFlags.ts` and `HAIOps.tsx` are not modified by this feature — gate logic is layered on top through provenance overrides only.

#### Enhanced AI Analysis Panel (`LayerAiPanel`)
Each layer (L1/L2/L3) has an AI Analysis panel with:
- **Pre-analysis data summaries**: L1 shows ADMET radar chart + target auto-match explanation + key metrics (IC50, Caco-2, hERG). L2 shows PK curve preview + clinical PK bars (Cmax, t½, responder %). L3 shows success probability gauge + trial metrics (power, p-value).
- **Safety Flag Gate**: Expandable `AiSafetyGate` component showing all safety flags (critical/major/warning severity) with ICH references and review acknowledgment button. Shows "clear to proceed" when no flags exist.
- **AI Report**: Claude-powered deep interpretation streamed via `/api/bio-analysis` endpoint with structured sections (overview, safety, target engagement, competitive context, etc.).

#### Simulation Architecture
All simulation engines are browser-side. Key engines include:
-   `admet.ts`: Rule-based ADMET scoring, organ risk, toxicity alerts.
-   `cohortSim.ts`: Two-compartment PBPK, NHANES population variability, ChEMBL Emax PD, FAERS AE calibration.
-   `trialSim.ts`: Statistical power, Weibull survival, interim analyses, cost estimation.

### System Design Choices
-   **Monorepo Structure**: `artifacts` for deployable apps, `lib` for shared libraries, `scripts` for utilities.
-   **TypeScript**: Extensive use with composite projects for improved type-checking and dependency management.
-   **Database**: PostgreSQL with Drizzle ORM for schema definition and interaction.
-   **API Design**: Express server with Zod for robust request/response validation.
-   **RDKit.js (MinimalLib)**: Used for molecular property calculations (`amw`, `CrippenClogP`, `lipinskiHBD`, `lipinskiHBA`, `tpsa`, `NumRotatableBonds`), pharmacophore detection, fingerprint generation, and similarity computation.
-   **Target Suggestion**: Two-tier approach using ChEMBL experimental data and RDKit SMARTS pharmacophore-based fallback.

### Python AI Service (`artifacts/ai-service/`)
A FastAPI-based Python backend service running on port 8090, proxied through the Node.js API server at `/api/ai/*`. Provides:
-   **ADMET Prediction** (`POST /api/ai/admet`): Full ADMET profiling — absorption (Caco-2, HIA, P-gp), distribution (BBB, PPB, Vd), metabolism (CYP inhibition, hepatic clearance), excretion (half-life), toxicity (LD50, hERG, Ames, DILI), and druglikeness (Lipinski, Veber, QED).
-   **Molecular Properties** (`POST /api/ai/properties`): Solubility, bioavailability, ionization, complexity, synthetic accessibility, lead/fragment-likeness assessment.
-   **Drug-Target Interaction** (`POST /api/ai/binding`): Three-path predictor — (1) curated pharmacophore QSAR for known reference targets EGFR/BRAF/BCL2/COX2/HDAC when target_id matches; (2) **real ChEMBL similarity-based prediction** with measured pIC50, UniProt accession, gene name, and PDB structure IDs (via `/api/chembl/similar-targets` route which aggregates activities from compounds ≥70% Tanimoto-similar, filters to protein targets via target-detail metadata, enriches via UniProt cross-references); (3) explicit empty result with status metadata when ChEMBL is unreachable — never fabricates predictions from a hardcoded list.
-   **Protein Analysis** (`POST /api/ai/protein`): UniProt + AlphaFold integration, amino acid composition, secondary structure prediction, druggability assessment.
-   **Full Pipeline** (`POST /api/ai/full-pipeline`): Runs all four modules in one call.
-   Frontend client library: `artifacts/hakase-ai/src/lib/aiService.ts`
-   Uses: Python 3.11, FastAPI, RDKit (full), scikit-learn, numpy, pandas

-   **ADMET-AI** (`POST /api/ai/admet-ml`): Open-source Chemprop-RDKit GNN ensemble (MIT license, github.com/swansonk14/admet_ai) trained on the TDC ADMET benchmark. -   **Implementation**: `artifacts/ai-service/models/admet_ml.py` (singleton, thread-safe), surfaced through `main.py` and proxied through `api-server/routes/ai-service.ts`.
-   **L1 UI integration**: New `MlAdmetCard` component (`artifacts/hakase-ai/src/components/MlAdmetCard.tsx`) is wired as the **default tab** ("ML-Enhanced") inside `Layer1Results` in `HakaseAI.tsx`. -   **DILI-ML (`POST /api/ai/dili-ml`)**: Dedicated DILI-risk binary classifier driving L3 Stage 2. Trained on TDC `DILI` (Xu et al. 2015, n=475 drugs labeled by DILIrank). Uses Random Forest (300 trees, balanced class weight) on Morgan fingerprint (1024-bit, radius 2) + 6 RDKit descriptors (MW, LogP, TPSA, HBD, HBA, RotatableBonds). 5-fold CV ROC-AUC = 0.881 ± 0.023. First call trains and pickle-caches to `artifacts/ai-service/cache/dili_ml_rf.pkl` (~5s); subsequent starts load from cache and prewarm in a background thread at boot (`artifacts/ai-service/main.py` startup hook), so warm inference is ~57 ms. Returns `{ dili_probability, risk_class (Low/Moderate/High at 0.30/0.55), severity_hint (none/mild/moderate/severe — purely heuristic banding of probability, NOT the DILIrank 5-class severity) }` plus an explicit `scope_caveat` string in the response stating that mechanism flags (mitochondrial / BSEP / ROS / immune) are NOT predicted. Implementation: `artifacts/ai-service/models/dili_ml.py` (mirrors `cardiotox.py`). Frontend: `predictDiliMl()` + `DiliMlResult` in `aiService.ts`. **L3 UI integration**: Stage 2 in `Layer3ExVivoScaffold.tsx` becomes a `wired:"ml-service"` stage with a "Run stage" button (alongside the existing Manual Entry toggle). The `Stage2Result` is now a discriminated union (`kind:"ml" | "manual"`); ML results render in a violet result card with prob / risk class / severity hint / training metadata; the Stage 6 handoff envelope tags ML-derived rows as `tier:"t0"` and manual-derived rows as `tier:"t-user"`. **Stage 5 biomarker derivation still requires Manual Entry** (mechanism flags) — the ML classifier cannot drive it. The Run-Full-L3-Pipeline button now auto-runs Stages 1 + 2; Stages 3–6 remain Manual-Entry stubs until Phases 2 & 3 ship.
-   **CardioTox (`POST /api/ai/cardiotox`)**: Dedicated hERG cardiotoxicity classifier that augments ADMET-AI's hERG with a methodologically independent second opinion. Trained on TDC `hERG` (Wang et al., 655 compounds, IC50 < 10 µM = blocker label) — distinct from ADMET-AI's `hERG_Karim` (different N, different curation). Uses Random Forest on Morgan fingerprint (1024-bit, radius 2) + 7 RDKit descriptors (MW, LogP, TPSA, HBD, HBA, FractionCSP3, basic-N count) — distinct algorithm class from ADMET-AI's Chemprop GNN. Reports 5-fold CV ROC-AUC (~0.81) with the prediction. First call trains and pickle-caches to `artifacts/ai-service/cache/cardiotox_rf.pkl` (~22s); subsequent server starts load from cache (~1s) and warm inference is sub-200ms. Returns `{ blocker_probability, risk_class (Low/Moderate/High), ic50_band }`. Implementation: `artifacts/ai-service/models/cardiotox.py`. Frontend: `predictCardiotox()` + `CardiotoxResult` in `aiService.ts`. **L1 UI integration**: `MlAdmetCard` fires `predictCardiotox` in parallel with `predictAdmetMl`; the hERG entry in Critical Safety Endpoints renders as a paired card showing both ADMET-AI hERG probability and CardioTox blocker probability + IC₅₀ band, with a Consensus / Aligned / Divergent badge based on risk-class agreement (same Low/Mod/High thresholds applied to both probabilities). Falls back gracefully if CardioTox is unavailable. **Note on RDKit**: a stale `rdkit-pypi 2022.9.5` was shadowing `rdkit 2026.3.1` and crashed under numpy 2.x; the legacy package was removed and `rdkit 2026.3.1` is now the single installed RDKit.
-   **AlphaFold per-residue pLDDT analysis (Target Protein Configuration)**: The Layer 1 Target Protein Configuration card now extracts per-residue pLDDT from AlphaFold's PDB file (B-factor column of CA atoms) in addition to the scalar `meanPlddt`. This unlocks structural quality classification (Excellent/Good/Fair/Poor) and segmentation into disorder regions (pLDDT < 50 for ≥10 contiguous residues) and well-folded domains (pLDDT ≥ 70 for ≥40 contiguous residues). For huge multi-domain targets (e.g. EGFR 1210 aa), the largest folded domain is surfaced as the **Druggable core domain** — letting users see that the actual druggable kinase region is ~residues 435–636 even though the full protein has a ~190 aa disordered C-terminal tail. Implementation is purely client-side in `proteinApi.ts` (`fetchAlphaFoldFull`, `fetchAlphaFoldPlddt`, `parsePlddtFromPdb`, `analyzePlddt`); the AlphaFold PDB download is small (~600 KB for a 1200 aa protein) and parsing is sub-10ms. Also fixed a stale `v4` URL fallback (current AlphaFold release is `v6`) and switched to the canonical `globalMetricValue` field for mean pLDDT. UI: Layer 1 → Target Protein Configuration result card now shows quality badge, confident/disordered fractions, folded-domain count, disorder-region count, and a highlighted druggable-core-domain box for proteins > 800 aa.
-   **ProteinML (`POST /api/ai/protein-ml`)**: ESM-2 t12 35M (`facebook/esm2_t12_35M_UR50D`, MIT, Meta AI Research) sequence-embedding druggability classifier that augments the rule-based protein druggability score the same way CardioTox augments hERG. Methodologically independent from `/api/ai/protein`: rule-based path scores binding-site / active-site counts and sequence length over UniProt features; ML path mean-pools the frozen ESM-2 residue embeddings (CLS + EOS excluded; sequences >1022 aa handled via **non-overlapping** chunking — chunks tile at stride == 1022 so each residue contributes to exactly one chunk-mean and the chunk-length-weighted average reduces algebraically to the true global per-residue mean; an earlier overlapping-stride implementation double-counted residues in the overlap zone and was corrected) and feeds a StandardScaler + Logistic Regression head. Training set is 161 hand-curated UniProt entries (81 well-drugged human targets across kinases / GPCRs / NRs / ion channels / proteases / transporters vs 80 essential housekeeping proteins with no approved direct binder; see `artifacts/ai-service/models/protein_ml_data.py`). Reports 5-fold CV ROC-AUC ≈ **0.87 ± 0.09**. First cold start fetches all 161 sequences from UniProt REST, embeds them, trains the head, and pickle-caches `artifacts/ai-service/cache/protein_ml_seqs.json` + `protein_ml_embeddings.npz` + `protein_ml_lr.pkl` (≈3 min cold; subsequent starts load from cache and re-train the LR head only). Class thresholds: ≥0.65 = "Druggable", ≥0.40 = "Borderline", else "Likely undruggable". Returns a CardioTox-style envelope `{ source: "ml" | "fallback", ml_available, sequence_length, prediction: { druggability_probability, druggability_class }, model_info: { ... } }`; UniProt lookup failures degrade to `source: "fallback"` at HTTP 200 so the consumer can rely on the rule-based score alone. Implementation: `artifacts/ai-service/models/protein_ml.py`. Frontend: `predictProteinMl()` + `ProteinMlResult` in `aiService.ts`; `protein_ml` block also exposed via `/api/ai/models/status` + `/api/ai/models/warmup`. **Target Protein Configuration UI integration**: `handleProteinFetch` in `HakaseAI.tsx` fires `predictProtein` and `predictProteinMl` in parallel with stale-request guards (mirroring the AlphaFold path). A new "Druggability assessment" paired card renders directly below the AlphaFold structural-quality block, showing the rule-based class + score on the left and the ESM-2 ML class + probability + CV ROC-AUC on the right, with an Aligned / Partial / Divergent agreement badge based on Druggable / Borderline / Undruggable bucket overlap. Either side degrades gracefully if its endpoint fails; the entire block is hidden if both sides have no data and nothing is in flight.
-   **SolubilityML (`POST /api/ai/solubility-ml`)**: Aqueous solubility (logS, mol/L) regressor that augments the rule-based Delaney ESOL formula already present in `predictProperties()` with a methodologically independent ML second opinion. Methodology independence is the core point: rule-based path is the closed-form Delaney 2004 equation `logS = 0.16 − 0.63·logP − 0.0062·MW + 0.066·RB − 0.74·AP` (no learning); ML path is a `RandomForestRegressor` (n_estimators=300) on a 1024-bit Morgan fingerprint (radius 2) concatenated with 9 RDKit descriptors (MW, LogP, TPSA, HBA, HBD, RotBonds, AromaticRings, AromaticProportion, FractionCSP3). Training set is 136 curated compounds blending the public Delaney 2004 ESOL dataset with literature/DrugBank measured logS values, spanning logS −8.94 to +1.57 (`artifacts/ai-service/models/solubility_ml_data.py`). Reports 5-fold CV **R² ≈ 0.74 ± 0.15, RMSE ≈ 0.96 logS** with the prediction. Class mapping for UI parity: `> -2 = Highly soluble`, `> -4 = Soluble`, `> -6 = Slightly soluble`, else `Insoluble` (rule-based side uses a slightly different 5-bucket scheme — `_sol_class` in `molecular_properties.py` — so the UI consensus badge compares the numeric Δ logS, not the class strings, which is more honest). First server start trains in ~5s and pickle-caches to `artifacts/ai-service/cache/solubility_rf.pkl`; subsequent starts load from cache and warm inference is sub-100ms. Returns `{ source: "ml" | "fallback", ml_available, prediction: { logS_predicted, solubility_class, solubility_mg_mL }, model_info: { ... } }`. Implementation: `artifacts/ai-service/models/solubility_ml.py`. FastAPI exposure + `/models/status` block + `/models/warmup` + background prewarm in `main.py`. Express proxy in `api-server/routes/ai-service.ts`. Frontend: `predictSolubilityMl()` + `SolubilityMlResult` in `aiService.ts`; `solubility_ml` block also exposed via `/api/ai/models/status`. **L1 UI integration**: A new "Molecule ML Predictions" card in `HakaseAI.tsx` (directly below Target Protein Configuration) hosts an ADMET-grouped paired-view scaffold; the first row, "Solubility & Permeability", renders rule-based logS (via `predictProperties`) and ML logS side by side with an Aligned (Δ < 0.5) / Partial (Δ < 1.5) / Divergent (else) consensus badge. The card scaffolds Distribution / Metabolism / Toxicity sub-headers as visible "More models coming" placeholders so future molecule-side models (CYP, Caco-2, PPB, CL_int, Ames, BBB) drop in without UI restructuring. Effect fires both predictions in parallel — first valid SMILES fires immediately for instant feedback, subsequent edits are debounced 400ms; each side's response is guarded by a `solSeqRef` so stale responses can never overwrite a newer query.
-   **BindingML (`POST /api/ai/binding-ml`)**: DeepDTA-style structure-aware binding affinity regressor that pairs the existing ChEMBL-similarity binding path (`/api/ai/binding`) with a methodologically independent ML second opinion. Methodology independence is the core point: ChEMBL path is a pure database lookup that aggregates measured pIC50 from ≥70%-Tanimoto-similar compounds and returns nothing when no neighbors exist; ML path is a learned `GradientBoostingRegressor` (n_estimators=300, max_depth=4, lr=0.05) on a 1512-dim feature vector that concatenates a 1024-bit Morgan fingerprint + 8 RDKit descriptors (drug, 1032-dim) with the **480-dim ESM-2 t12 35M mean-pooled per-residue embedding** of the UniProt sequence (target, 480-dim). The ESM-2 model is **shared with ProteinML** via `models.protein_ml._embed_sequence` / `_fetch_sequence` so the ~140 MB weights and the embed cache are loaded exactly once across both heads. Training set is **116 curated literature pKd pairs spanning 56 unique UniProts and 61 unique drugs** (v1.1, after architect honesty pass — 14 placeholder/family-archetype-SMILES rows from v1.0 were removed and replaced with 6 verified literature pairs across BRAF/VEGFR2/PSMB5/COX1, plus an Imatinib/EGFR off-target measurement) covering kinases (ABL1/SRC/EGFR/BRAF/JAK/CDK/MEK/BTK/VEGFR2), GPCRs (β-blockers, DRD2, HTR2A, OPRM1), COX1/2, HDAC1-3, PARP1/2, BCL2, ACE/ARB, statins, SERT, GABA-A, PI3Kδ, aromatase, ER, PDE5, and proteasome — sources tagged "Karaman2008" / "Davis2011" / "BindingDB" / "DrugLabel" / "Lit" (`artifacts/ai-service/models/binding_ml_data.py`). Reports **two CV regimes side-by-side** so users see both upper- and lower-bound generalization honestly: **random 5-fold KFold R² ≈ 0.16 ± 0.18, RMSE ≈ 1.08 logKd, Pearson r ≈ 0.53 ± 0.15** (in-distribution, same target may appear in train+test) AND **GroupKFold-by-UniProt R² ≈ 0.16 ± 0.23, RMSE ≈ 1.06** (leave-target-out, novel-target lower bound — comparable random vs grouped R² is a good sign the model isn't merely memorizing target identity). The frontend `CV R² 0.16±0.18 · grp R² 0.16±0.23` badge surfaces both numbers with their ±std rather than hiding the small-N uncertainty. Embed cache (`cache/binding_ml_embeddings.npz`) is hardened with **`EMBED_CACHE_VERSION="esm2_t12_35M_v1"` sentinel + dim validation + atomic tmp+fsync+os.replace writes + per-process lock**; NaN-failed embeddings are NOT persisted; cache misses without the sentinel auto-rebuild on next start. `MODEL_VERSION="1.1.0"` invalidates the v1.0 pickle automatically. Class mapping for UI badge: `pKd ≥ 8 = High affinity (Kd ≤ 10 nM)`, `≥ 7 = Moderate (≤ 100 nM)`, `≥ 6 = Weak (≤ 1 µM)`, else `Negligible` — the L1 UI displays a Δ-based consensus badge (Aligned <0.5, Partial <1.5, Divergent ≥1.5) when both ChEMBL and ML return a number, and **branches three distinct empty states off `bindingChembl.model_info.status`**: `service_unreachable` → amber "ChEMBL service unreachable — try again later" (infrastructure failure, distinct from biology), `no_targets` / `requested_target_not_found` → "No ChEMBL neighbors above 70% — ML side only" (true biological signal), and an "No ChEMBL data returned" fallback for any other empty case — so the user can never confuse a ChEMBL outage with a real "no near-neighbors" finding. Verified post-retrain (Apr 2026): Imatinib+ABL1 → pKd 9.08 (High), Erlotinib+EGFR → pKd 9.10 (High), Caffeine+EGFR off-target → pKd 6.59 (Weak — correct discrimination on a non-kinase ligand). Reference implementation cites Öztürk et al. 2018 DeepDTA. ESM-2 weights MIT (Meta AI Research). The honest data. First server start trains in ~50s on a warm ESM-2 cache (~5s on a fully warm restart) and pickles the bundle to `artifacts/ai-service/cache/binding_ml_gbm.pkl`; per-target embeddings live in `artifacts/ai-service/cache/binding_ml_embeddings.npz`. Returns `{ source: "ml" | "fallback", ml_available, prediction: { pKd_predicted, Kd_nM, affinity_class }, model_info: { ... } }`. Implementation: `artifacts/ai-service/models/binding_ml.py`. FastAPI exposure at `/predict/binding-ml` + `/models/status` block + `/models/warmup` + background prewarm thread in `main.py`. Express proxy in `api-server/routes/ai-service.ts`. Frontend: `predictBindingMl()` + `BindingMlResult` in `aiService.ts`; the **paired-view "Binding affinity" card** in `HakaseAI.tsx` lives directly under the Druggability assessment paired card in the Target Protein Configuration section, fires both ChEMBL and ML predictors in parallel on `(smiles, uniprotId)` change with a 400 ms debounce + monotonic stale-request guard, and renders only when both inputs are present and the SMILES is RDKit-valid.
