## 2. organoid-response-ml — concrete bootstrap and minimum-N

Fair criticism, this was hand-wavy. Concrete numbers:

- **Day-1 bootstrap cohort, realistically:** 50–150 compound-organoid pairs assembled from HCMI published validation studies, the PDXE encyclopedia (Novartis 2015 — predates organoid era but the efficacy data is comparable), Tox21 organoid-mode expansions since 2023, and supplementary data from major HUB-line validation papers. Heavily oncology-skewed. DepMap PRISM gives 6000 × 2000 IC50s but those are 2D cell lines, useful only as labelled-proxy with an explicit "2D not organoid" tag.
- **Minimum-N to ship the calibration model:** 30 compound-organoid pairs per organoid type, OR cohort spans ≥3 chemical scaffolds. Below that, the model is suppressed and we surface the structural prediction (cellTypeAggregator output) alone with its un-calibrated band.
- **Retraining trigger:** not weekly. Trigger when calibration cohort grows by ≥10% OR observed fold-error band shifts by ≥1.2× — whichever comes first. Steady-state will probably be monthly, not weekly.
- **Ownership:** Performance & Lineage Swarm (per the HAIOps playbook) owns the retraining pipeline. Compliance Swarm owns the QMRF refresh per retrain. Each retrain bumps the model_version and emits a cascade flag 6 event with the new fold-error band.

## 4. `prediction_only` as a discriminated union, not a string

Agreed, this needs to be a tagged enum branch in the envelope type so consumers must handle it explicitly:

```typescript
type CascadeFlagState =
  | { kind: 'pass' }
  | { kind: 'fail'; reason: string; remediation: RemediationRoute }
  | { kind: 'prediction_only'; method: string; foldError: FoldErrorBand;
      applicabilityDomain: { inDomain: boolean; score: number };
      snapshotsWithinCadence: boolean };

// Curators / consumers force exhaustive handling:
function evaluateForSigning(s: CascadeFlagState): SignatureDecision {
  switch (s.kind) {
    case 'pass': return canSign;
    case 'fail': return blockWithReason(s.reason);
    case 'prediction_only': return requireWetAnchor(s.method, s.foldError);
    default: const _exhaustive: never = s; return _exhaustive;
  }
}
```

Amber Curator's refusal-to-sign becomes a function whose type signature accepts only `{ kind: 'pass' }` or a specific approved subset for load-bearing fields. Compile-time guarantee, not runtime business logic. Updating the spec.

## 5. T1 retrieval — real spec

Was understated. Concrete additions:

- **Similarity gates:** Tanimoto ≥ 0.85 = "very similar analog" (high-confidence retrieval); 0.7–0.85 = "near analog" (informational); < 0.5 = no match returned. Two thresholds, surfaced separately so the user can see which bucket a hit falls into.
- **Distinct typed errors:** `NoAnalogsFound` (clean empty state, similarity scan completed) vs. `ServiceUnreachable` (ToxRefDB or MPS-Database is down — we don't know if analogs exist) vs. `RegistrationRequired` (the API needs auth we haven't completed). Each is a different envelope state, not a single "no data" string.
- **ToxRefDB friction:** registration-gated, finicky API, schemas have shifted. Realistic ingest is closer to 1.5–2 weeks. Plus MPS-Database which is small and the schema changes — their team is responsive but we have to build to a moving target.
- **Pattern match to existing code:** mirror the ChEMBL similar-targets v2 retrieval path that already works at L1. Same Tanimoto computation, same caching, same provenance discipline.

T1 stage estimate moves from 5–7 days to **~2 weeks, parallel to Stages 3–6**.
