## The ChEMBL API Fix — What You Actually Need to Build

You identified the precondition correctly. Here's the specific contract:

**Current API returns:**
```
bestValue_nM: 1.2
bestActivityType: "IC50"
bestPChembl: 8.92
bestSimilarity: 0.87
```

**Required API returns:**
```
bestValue_nM: 1.2
geometricMean_nM: 3.4
range_nM: [0.4, 18.0]
nMeasurements: 8
species: ["human", "human", "mouse"]  // or humanOnly: false
assayTypes: {"IC50": 5, "Ki": 2, "Kd": 1}
assayFormats: {"biochemical": 6, "cell-based": 2}
confidence: "high" | "medium" | "low"  // derived from n + spread + species consistency
```

**UI rendering rule:**
- If `confidence === "high"` AND `species === ["human"]` AND `assayTypes` has one dominant type: show point estimate with small print.
- If `confidence === "medium"`: show range as primary, point estimate as secondary.
- If `confidence === "low"` OR `nMeasurements < 3`: show "insufficient data — estimate suppressed" with link to raw ChEMBL records.

This is not a prediction problem. This is a **data quality signaling** problem. The fix is in the API aggregation logic and the UI rendering layer, not in a new ML model.
